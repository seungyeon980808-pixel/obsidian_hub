## 2. 핵심 개념 — 한 줄 요약

```
작성   → 맥 / 윈도우 / 모바일 어디서든 (전부 Firestore로 모임)
내보내기 → 윈도우에서만 (= 학교 PC가 vault 본진)
내보내면 → 보관함이 비워진다 (성공한 것만 삭제)
```

선택·제외 없음. **전부 모았다가 전부 비운다.** 애매하게 고를 거면 다른 앱을 쓴다.

## 3. 기능 명세

### 3-1. 메모 작성 (수집)
- 첫 줄에 제목처럼 한 줄, 그 아래 본문 — 이게 기본 작성 패턴
- 작성 즉시 Firestore에 저장
- `[[링크]]`, `#태그` 등 Obsidian 문법은 **본문에 그대로 보존**

### 3-2. 동기화 상태 표시 (모바일 유실 방지)
- 메모마다 동기화 상태 아이콘 표시
- 판정 기준: Firestore 스냅샷의 `hasPendingWrites` 메타데이터
  - `false` → `✓ 저장됨` (서버 도달 확인)
  - `true`  → `⚠ 대기 중` (로컬 캐시에만 있음, 오프라인)
- **중요**: Firestore는 오프라인에도 `addDoc()`이 성공처럼 응답함.
  순진하게 "저장 완료"를 띄우면 안 되고, 반드시 `hasPendingWrites`로 서버 도달을 구분할 것
- 닫기 경고(`beforeunload`)는 **넣지 않음**
  - 모바일 웹은 보통 탭을 닫지 않고 백그라운드로 내림 → 로컬 캐시 유지됨 → 유실 위험 낮음
  - `beforeunload`는 모바일에서 잘 동작하지도 않음

### 3-3. 내보내기 (export)
- 윈도우 데스크탑 크롬에서만 실행
- **전부 내보내기** — 보관함 전체가 대상. 부분 선택/제외 없음
- 실행 전 **확인창 1회** 표시:
  - 내보낼 메모 목록 미리보기
  - "내보낸 메모는 보관함에서 삭제됩니다" 경고 문구
  - [취소] / [내보내기]
- **순서 절대 엄수 (유실 방지의 핵심)**:
  ```
  1. 메모를 .md 파일로 폴더에 쓴다
  2. 쓰기 성공을 확인한다
  3. 확인된 메모만 Firestore에서 삭제한다
  ```
  - 절대 "삭제 먼저 → export" 금지
  - 메모 단위로 처리. 5개 중 3개 성공 시 → 성공한 3개만 삭제, 실패한 2개는 보관함에 남김

### 3-4. 폴더 권한 유지
- 폴더 핸들을 IndexedDB에 저장
- 재방문 시 "이 폴더 계속 쓸까요?" 클릭 1회 → 바로 작업
- 브라우저 정책상 완전 무인 자동화는 불가 (클릭 1회는 불가피)

## 4. 파일명 생성 로직

평소엔 깔끔한 이름, 문제 상황에서만 시간이 끼어든다.
(`[[링크]]`가 파일명을 참조하므로 파일명은 깔끔해야 백링크·그래프뷰가 의미 있음)

```
1. 첫 줄 추출 → 앞뒤 공백 제거
2. 파일명용 특수문자 제거: \ / : * ? " < > |
   - [[ ]], # 등은 파일명에서만 제거, 본문에서는 보존
   - ※ 정리는 '파일명 만들 때만'. 본문 내용은 절대 건드리지 않음
3. 길이 제한: 50자에서 자름
4. fallback: 위 결과가 빈 문자열이면 → 날짜+시간 (예: 2026-06-15_1432)
5. 중복 방지: 폴더에 같은 이름이 이미 있으면 → 뒤에 시간 (예: 회의록_1432)
```

- 같은 첫 줄("회의록")로 또 쓴 메모는 **무조건 별개 파일**로 저장 (`회의록_1432.md`)
- 기존 파일에 합치는 동작은 하지 않음 (합치기는 Obsidian 안에서 직접)

### YAML frontmatter
각 .md 파일 상단에 자동 생성:
```yaml
---
created: 2026-06-15 14:32
tags: []
status: inbox
---
```

## 5. Firebase 설계

- **edunote-96bd7과 분리된 전용 신규 프로젝트** 사용
  - 이유: edunote는 학생 315명 데이터가 도는 실사용 앱. 개인 메모와 물리적으로 격리
  - 무료 한도도 프로젝트별로 따로 받음 (edunote 한도와 안 겹침)
  - 신규 프로젝트라 기존 edunote/퍼스널 사이트에 영향 0
- 신규 프로젝트의 config 값을 코드에 정확히 넣을 것 (edunote config 복붙 금지)
- 구글 로그인은 신규 프로젝트에서 처음 1회 다시 인증 필요
- Firestore 경로 및 보안 규칙:
  ```
  match /users/{uid}/inbox/{doc} {
    allow read, write: if request.auth.uid == uid;
  }
  ```
  - 본인 UID 것만 read/write

## 6. 기기 구성

| 기기 | 역할 |
|---|---|
| 모바일 | 작성 (수집). 통근 중 메모 |
| 맥 (집) | 작성만 |
| 윈도우 (학교) | 작성 + **내보내기 (vault 본진)** |

- vault는 **윈도우 로컬 폴더 한 곳**에만 존재 → 기기 간 vault 동기화 불필요
- vault는 **순수 로컬 폴더** (구글 드라이브 안에 두지 않음)
  - 드라이브 안에 두면 export 직후 동기화 충돌/충돌 사본 위험
- 집에서 쓴 메모는 학교 PC에서 내보내기 전까지 Firestore에만 떠 있음 (정상 동작)

## 8. 핵심 위험 요약 (개발 시 반드시 의식)

| 위험 | 대응 |
|---|---|
| export 도중 실패 → 메모 유실 | 쓰기 성공 확인 후에만 삭제. 실패 메모는 남김 |
| 오프라인인데 "저장됨" 거짓 표시 | `hasPendingWrites`로 서버 도달 구분 |
| 같은 제목 메모 덮어쓰기 | 빈값/중복 시 시간 붙임. 무조건 별개 파일 |
| 첫 줄이 빈 줄/특수문자뿐 | 날짜+시간 fallback |
| 학생 데이터와 섞임 | edunote와 별도 Firebase 프로젝트 |
| vault가 드라이브 안 → 충돌 | 순수 로컬 폴더에만 vault 둠 |

## 9. 개발 순서 (제안)

1. 신규 Firebase 프로젝트 생성 + Firestore + 구글 로그인 설정
2. 작성/목록 UI (모바일 우선) + Firestore 저장
3. `hasPendingWrites` 기반 ✓/⚠ 상태 표시
4. 파일명 생성 로직 (fallback·중복 처리 포함)
5. 내보내기: 확인창 → 폴더 쓰기 → 성공 확인 → 삭제 (순서 엄수)
6. 폴더 핸들 IndexedDB 저장 (재방문 권한 유지)
7. YAML frontmatter 생성
8. 디자인 다듬기 + 버전 표기
